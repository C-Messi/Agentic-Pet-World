# Pet 小镇扩展设计

## 1. 目标

在现有 Agent Cat House 的像素猫屋之外，增加一个与猫屋平级、可持久演化的 Pet 小镇。桌宠可以自主提出外出，用户也可以主动放行；进入小镇后，桌宠继续由 Agent 与规则系统驱动，不提供方向键式直接移动。

首版体验需要同时成立：

- 小镇中存在 4 只性格不同的常驻桌宠。它们由本地 Mock 数据驱动，但主画面只显示角色名字，不用破坏沉浸感的 Mock 标签，也不宣称背后存在真实在线用户。资料页统一称为“小镇常驻居民”。
- 桌宠能与居民相遇、交流、嬉闹、共同抽签、摆摊和改造小镇。
- 桌宠的经历来自真实、可回放的小镇事件。用户观看小镇时能看到关键片段；桌宠回家后会主动讲述，并生成可回看的见闻卡。
- 用户关闭页面后不逐秒运行后台 Agent。再次打开时，系统按离开时长生成少量受约束的摘要事件，并回放关键片段。
- 首个可扩展 AI 活动是签亭。未来的 Agentic Game 使用相同活动契约接入。
- 小镇继续使用现有温暖、低分辨率、清晰轮廓的像素画风，并保持 Phaser 全屏世界为第一视觉层级。

真实联机、自由生成建筑、完整经济系统、持续后台 Agent 和第二个完整娱乐活动不属于首版。

## 2. 产品体验

### 2.1 外出与回家

采用双入口外出：

1. 桌宠可以依据性格、当前目标、关系和冷却时间主动提出外出意愿。
2. 用户也可以随时发起“放出去”操作。
3. 桌宠通过猫屋门口离场，客户端从 `WorldScene` 切换到持久的 `TownScene`。
4. 用户可以召回桌宠，但召回会等待不可中断的短活动阶段结束，避免截断事件状态机。
5. 桌宠也可以自主决定回家。回家后主动复盘本次旅程，并在存在值得保存的经历时生成见闻卡。

用户在猫屋和小镇中都通过自然语言与桌宠交流。切换场景不创建新的对话身份，也不清空已有记忆。

### 2.2 观察体验

小镇采用“混合观察”布局：

- Phaser 小镇画面占满主视口，默认相机跟随自家桌宠。
- 用户可点击居民或使用视角控件切换跟拍对象，但不能直接控制其移动。
- 普通行为只通过动画和短气泡表现；关键事件额外显示一行短字幕。
- 完整镇史、居民关系和见闻卡放在 React 抽屉中，不设置常驻事件侧栏。
- 底部自然语言命令栏沿用猫屋交互模式，并根据当前场景调整占位提示。

这套镜头语言提供“用户是观众、桌宠是生活主角”的体验，同时保留自然语言交流能力。

### 2.3 小镇布局

首版小镇包含以下区域：

- 中央广场：居民相遇、闲聊和嬉闹的主要公共空间。
- 签亭：首个完整 `TownActivity`，支持自主或受托抽签、结伴抽签与解签。
- 商摊街：居民展示宠物或主人个性的空间，支持摆摊、交流和宣传，不发生交易。
- 公共花园：预设的共同种植活动与轻量世界改造示例。
- 共创地块：允许铺路、种花、摆设、搭小摊和布置许愿角等受限建造。
- 游乐屋：未来 Agentic Game 的可发现入口；首版只表现为尚未开放的设施。
- 小镇门口：猫屋与小镇的视觉化切换点。

地图必须为每个活动区提供稳定区域 ID、可达入口、容量限制和交互点。地图变化不能堵塞关键路径、小镇门口或活动入口。

## 3. 系统架构

采用事件驱动的小镇引擎。浏览器负责实时表现，服务端维护世界真相，模型只能提出受约束意图。

### 3.1 浏览器职责

- `TownScene`：加载地图、碰撞、像素素材、相机跟拍、气泡和昼夜表现。
- `TownEntitySystem`：管理自家桌宠和常驻居民的精灵、导航、动画和占位。
- `TownEventPlayer`：按顺序播放服务端批准的移动、对话、嬉闹、抽签、摆摊和建造事件，并报告结果。
- React UI：提供家/镇切换、放行、召回、视角选择、镇史、关系、公开展示资料和见闻卡。

客户端不能自行提交持久世界改造。服务不可用时只允许播放缓存世界上的本地待机动画。

### 3.2 服务端职责

- `TownSimulationService`：推进在线节拍和离线摘要模拟，选择候选行为，验证参与者、区域容量、资源、冷却和路径约束。
- `TownEventStore`：持久化居民状态、关系、小镇事件、世界改造、活动状态、外出记录和见闻卡来源。
- `TownActivityRegistry`：注册签亭、商摊和未来娱乐活动的状态 schema、工具、区域和结果事件。
- `TownNarrator`：仅在关键决策、居民对白、签文解读、宣传语和见闻总结时调用模型。
- `TownProjectionService`：从事件流生成当前小镇快照、居民关系、镇史和可恢复的活动状态。

模型失败不停止小镇基础行为。规则系统仍可选择散步、休息、嬉闹、预设抽签解释和预设建造。

### 3.3 稳定契约

`TownResident` 在首版描述两类来源：

- `player-pet`：当前用户的桌宠。
- `resident`：首版常驻居民，由 Mock 配置和规则行为驱动。

来源字段保留可版本化的扩展位置，但首版 schema 不接受远程桌宠。未来真实联机需要通过新协议版本增加来源类型，客户端不能自行指定居民来源。

`TownIntent` 是规则系统或模型提出的行为提案，包含行为类型、参与者、目标区域、可选活动或建造配方以及因果来源。Intent 不直接修改状态。

`TownEvent` 是通过验证后产生的不可变事实，至少包含事件 ID、会话 ID、事件类型、参与者、地点、因果 ID、时间戳和有界 payload。所有投影必须能够通过事件重建。

`TownActivity` 声明活动 ID、标题、区域 ID、版本化状态 schema、初始状态、容量、允许工具、状态转移和结果事件。它借鉴现有 `MiniGameManifest`，但面向持续小镇活动，不与暂停世界场景的小游戏生命周期混用。

## 4. 小镇行为与改造

### 4.1 行为选择

每只居民同一时间只能参与一个活动。候选行为由以下输入决定：

- 居民性格与偏好；
- 当前关系和共同记忆；
- 未完成目标；
- 区域容量与其他居民位置；
- 行为冷却；
- 小镇当前设施与改造状态；
- 签运等软性叙事倾向。

普通节拍优先使用确定性加权规则。只有需要生成有意义的选择或文本时才调用模型，且模型返回的 Intent 必须经过共享 Zod schema 和服务端语义验证。

### 4.2 受限建造

首版提供版本化建造配方，例如铺设石板路、种花、摆放路灯、搭建摊位和布置许愿角。每个配方声明：

- 可用地块和占用网格；
- 必要的前置设施或事件；
- 允许参与人数；
- 构建阶段和持续时间；
- 像素素材 frame；
- 碰撞变化；
- 产生的 `TownEvent` 类型。

模型只能选择配方、位置和参与者，不能生成任意地图 JSON、素材路径、代码或碰撞数据。服务端拒绝越界、重叠、阻断关键路径或超出配额的改造。

每次改造产生事件并进入镇史。客户端刷新、重连和离线补演必须通过幂等事件 ID 避免重复建造。

## 5. 活动系统

### 5.1 签亭

签亭状态机为：

`idle -> gathering -> drawing -> revealed -> completed`

允许的活动工具包括邀请同伴、提出问题、抽签、展示签文、解签和结束活动。用户可以委托自家桌宠代抽；桌宠也可以因为自己的困惑、关系或小镇事件自主起念，并邀请其他居民共同参与。

签文来自版本化本地签池，包含稳定 ID、签级、签文、基础解释和允许的叙事主题。抽取使用可记录的种子，保证结果可复现。模型只能结合签文本义、参与者关系和已发生事件生成解读，不能改变签文结果或编造参与者。

签运只调整后续候选话题和行为权重，不强制控制行为，不预测现实结果，也不输出医疗、财务、法律、灾祸等高风险断言。模型不可用时使用基础解释模板完成活动。

### 5.2 人格商摊

商摊是 `TownActivity`，不引入货币、价格、库存、购买或利润。活动闭环为：

1. 桌宠自主申请摊位，或接受用户的摆摊委托。
2. 从空闲摊位、开放时长、预设主题、招牌样式和 1 至 3 个展示位中选择布置。
3. 桌宠生成有长度限制的宣传语并招呼路过居民。
4. 访客可以围观、提问、夸赞和回应。
5. 活动结束后，互动转为关系变化、镇史和可选见闻卡来源。

宠物性格、偏好和已有小镇经历可以直接用于摊位布置。主人的作品、兴趣、介绍或链接只有在用户明确加入“公开展示”资料后才能使用。系统不得从私密对话、长期记忆或模型推断中自动提取主人信息用于宣传。

公开展示资料需要独立 schema、显式增删界面、长度和数量限制。首版只渲染文本与预设像素图标，不支持上传任意媒体或可点击外部链接。

## 6. 主动叙事与关系

主动情绪价值来自桌宠自己的生活，而不是先依赖用户上下文推断情绪再安慰。

叙事链必须遵循：

`主动起念 -> 已验证事件 -> 关键直播 -> 回家讲述 -> 见闻卡`

用户观看小镇时，只将关键对话、共同活动、关系变化和世界改造提升为字幕事件。普通移动和待机不进入高优先级事件流。

关系是慢变量。共同活动、帮助、成功合作、分歧和和解只产生小幅、有上下限的关系变化。单次模型对白不能直接覆盖关系值。关系投影保留共同事件引用，使桌宠日后能够基于真实共同经历交流。

见闻卡包含标题、地点、参与者、事件引用、桌宠第一人称讲述、可选像素缩略图描述和创建时间。卡片只能引用持久化事件；Narrator 不能添加未发生的角色、行为或改造。没有值得保存的事件时可以只进行简短回家对白，不强制生成卡片。

## 7. 离线补演

关闭页面时记录最后确认的服务端时间和居民状态。用户回来后，服务端根据离开时长计算有上限的离线预算：

- 不逐秒补跑，也不声称世界始终实时运行。
- 每次恢复生成 0 至 5 个摘要事件；短暂离开可以不产生事件。
- 事件仍需通过与在线行为相同的容量、关系、活动和建造验证。
- 离线期间最多产生一次持久世界改造，避免用户回来后地图不可辨认。
- 不在离线补演中自动公开新的主人资料。
- 恢复响应包含起始投影版本、事件列表和最终投影版本，客户端按序播放关键片段后进入实时状态。

重复恢复请求使用恢复窗口 ID 保证幂等。若事件回放失败，客户端直接加载最终权威快照，并在镇史中保留事件记录。

## 8. 持久化与 API

在现有 SQLite 迁移体系中增加以下表：

- `town_residents`：居民来源、性格配置和当前状态。
- `town_events`：不可变小镇事件及因果关联。
- `town_relationships`：居民关系投影和最后应用的事件版本。
- `town_world_states`：小镇地图投影、改造状态和版本。
- `town_activity_instances`：活动类型、参与者和版本化状态。
- `town_outings`：外出、回家、最后确认时间和离线恢复窗口。
- `town_experience_cards` 与 `town_experience_card_events`：见闻卡及来源事件。
- `public_showcase_items`：用户显式维护的公开展示资料。

API 至少覆盖：获取小镇快照、放行、召回、推进/拉取事件、提交事件播放结果、执行离线恢复、读取镇史与见闻卡、维护公开展示资料。所有写接口都绑定服务端会话，验证幂等键和投影版本；客户端不能指定居民来源或直接写入关系、地图和见闻卡。

事件 payload 和活动状态必须为有界、版本化、JSON 兼容数据。迁移需要保留现有猫屋会话和对话数据。

## 9. 失败处理与安全

- 模型超时、不可用或返回无效输出：使用确定性候选行为和模板文本，保留小镇基础可玩性。
- 服务端离线：显示缓存快照和本地待机动画；禁止持久事件、关系变化、建造、抽签结果和摆摊资料修改。
- 事件播放失败：上报结构化结果，停止当前播放链，并从权威快照恢复。
- 无效 Intent：记录诊断信息并拒绝，不能部分修改世界。
- 重复请求：通过事件 ID、恢复窗口 ID 和活动实例版本幂等处理。
- 活动场景加载失败：保持小镇场景可用，桌宠离开活动点，并显示可重试状态。
- 所有模型输出继续使用共享 Zod schema；未知居民、区域、活动、配方、工具和事件类型一律拒绝。
- Mock 常驻居民不显示技术标签，但产品也不展示“真人在线”“其他用户正在玩”等虚假声明。

## 10. 验证与验收

### 10.1 自动化验证

- 单元测试：共享 schema、Intent 验证、事件 reducer、投影幂等、关系变化、行为权重、建造占用、活动注册、签文抽取、公开资料边界和离线预算。
- 服务端集成测试：事件持久化、投影重建、恢复窗口幂等、模型降级、世界改造恢复、活动状态迁移和旧会话兼容。
- 客户端测试：场景切换、事件顺序播放、跟拍选择、气泡布局、字幕优先级、权威快照恢复和抽屉状态。
- E2E：放行进入小镇、切换跟拍、与常驻居民互动、共同抽签、完成一次建造、开设人格商摊、回家讲述并打开见闻卡。
- 视觉验证：桌面与移动端的像素缩放、相机取景、活动区可见性、气泡、字幕、命令栏、抽屉和触控目标无重叠。

### 10.2 首版验收场景

使用确定性 Fake Provider 启动新会话后，用户可以放行桌宠进入小镇，看到至少 4 名常驻居民进行独立待机和社交行为。桌宠能够完成一次结伴抽签、一次受限建造和一次人格商摊活动。关闭并重新打开页面后，小镇改造保持存在，系统生成不超过 5 个可信离线事件。桌宠回家后主动讲述一段引用真实事件的经历，并生成一张可回看的见闻卡。模型不可用时，上述核心流程仍可通过模板和规则系统完成。
